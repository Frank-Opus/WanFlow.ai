## Design Context

### Users
- Primary users are Chinese-first evaluation operators, benchmark engineers, delivery managers, and data-company platform teams.
- They use this product to ingest heterogeneous source material, normalize it into a stable evaluation contract, manage `Project` / `SourceFile` / `ProblemItem`, run synchronous or asynchronous benchmarks, inspect evidence, and export client-ready artifacts.
- The interface must support enterprise handoff and internal operations, not a one-off demo flow.

### Brand Personality
- Enterprise-grade, disciplined, trustworthy.
- Chinese-first with bilingual support.
- Serious infrastructure rather than consumer AI toy.
- The experience should evoke confidence, auditability, and operational control.

### Aesthetic Direction
- Editorial-enterprise command surface on a bright, restrained canvas.
- Clear hierarchy, high information density, and crisp containment inspired by enterprise documentation and control planes.
- Use cool neutrals, controlled blue interaction accents, and only limited warm signal accents for status or emphasis.
- Avoid generic AI-dashboard tropes: giant decorative metrics, excessive gradients, glass-heavy cards, duplicated intros, and oversized headlines carrying long paragraphs.

### Design Principles
1. Put operational priority first: run center, results, preview, and admin workflows must appear in the order users actually work.
2. Make evidence readable: source, answer, validation, latency, and artifacts should be easy to scan without jumping across disconnected surfaces.
3. Keep hierarchy tight: long copy belongs in body text, not in oversized headings; headings should be short and directional.
4. Use one coherent shell: navigation, panels, typography, radii, and spacing must feel like one enterprise product, not multiple experiments.
5. Preserve extensibility: the UI must continue to represent the fixed platform objects `Project`, `SourceFile`, `ProblemItem`, `BenchmarkRun`, and `Artifact` without collapsing back into a single-item demo.
